# 📦 Kit Tracker

Scan equipment out to jobs and back in again — so nothing gets left behind.

One person sets up gear at a client site and scans each item's QR label to
build the job's **kit list**. Whoever collects afterwards opens the same job,
scans items back in, and sees live exactly what's collected and what's still
out there. Anything unscanned when collection is closed gets flagged
**Missing** and shows up on the cross-job missing-items report.

## Quick start

```bash
cd kit-tracker
python -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate
pip install -r requirements.txt
python app.py
```

Open http://localhost:5000. The database (`instance/kit_tracker.sqlite`) is
created and seeded automatically on first run with demo equipment and one
in-flight demo job, so every screen is testable immediately:

- **Setup scan** — the demo job is Live; you can still add items to it.
- **Collection scan** — the demo job has 7 items on its kit list to tick off.
- **Inventory** — 12 seeded items; open any of them to print a QR label.

To start fresh, delete `instance/kit_tracker.sqlite` and restart.

## ⚠️ Camera access needs HTTPS

Browsers only expose the camera in a *secure context*:

- `http://localhost:5000` on your dev machine — **works** (localhost is exempt).
- `http://192.168.x.x:5000` from a phone on your LAN — **camera blocked**.

Options for testing the scanner on a real phone:

1. **Tunnel (easiest):** `ngrok http 5000` (or Cloudflare Tunnel / Tailscale
   Funnel) and open the HTTPS URL on the phone.
2. **Ad-hoc TLS:** `pip install pyopenssl`, then
   `flask run --host=0.0.0.0 --cert=adhoc` and accept the self-signed
   certificate warning on the phone.
3. **No camera needed:** every scan page also has a manual-entry box — type
   `KIT-0042` (or just `42`) to simulate a scan from any browser.

## How it works

- Each equipment item gets a QR label encoding its unique ID (e.g.
  `KIT-0042`), generated by the app (Inventory → item → Label → Print).
- **Setup mode:** each scan adds the item to the job's kit list and sets the
  item's status to *Out on Job*. Re-scanning shows "already added".
- **Collection mode:** the expected kit list is shown with live progress;
  each scan marks the item collected and sets it back to *In Warehouse*.
  Scanning something that isn't on the job's list warns you clearly.
- **Mark collection complete:** anything still unscanned is flagged
  *Missing* and the job becomes *Items Missing* (or *Completed* if all
  items came back). A missing item scanned back in later clears itself, and
  the job flips to *Completed* once nothing is left outstanding.
- **Staff identity:** a simple name dropdown recorded against every scan —
  no accounts or passwords (by design).

## Job lifecycle

```
Setup in Progress ──(mark setup complete)──▶ Live
        │                                     │
        └────────────(first collection scan)──┴──▶ Collection in Progress
                                                        │
                                     (mark collection complete)
                                                        │
                                        all collected ──▶ Completed
                                        some unscanned ─▶ Items Missing
                                                        │
                                       (late scan of last missing item)
                                                        └─▶ Completed
```

## Stack

Python · Flask · SQLAlchemy · SQLite · Jinja2 · vanilla JS ·
[html5-qrcode](https://github.com/mebjas/html5-qrcode) (CDN) for camera
scanning (works on iOS Safari, unlike the native BarcodeDetector API) ·
[qrcode](https://pypi.org/project/qrcode/) for label generation.

Out of scope for now (deliberately): accounts/auth, offline support,
email/SMS alerts, multi-warehouse support.
